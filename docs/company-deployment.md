# WorkHub + Hermes + Feishu Company Deployment

This is the deployment runbook for a new company computer, a new operator, or an AI agent that did not participate in the original implementation.

Read this document together with `AGENTS.md` before deploying or changing the Hermes integration.

## What Is Being Deployed

```text
Feishu Bot
  -> Hermes Agent
  -> WorkHub MCP server (scripts/hermes-workhub-mcp.mjs)
  -> WorkHub HTTP endpoint (/api/integrations/hermes/workhub)
  -> WorkHub database
```

WorkHub is the structured system of record. Hermes interprets natural language and calls MCP tools. WorkHub does not call an LLM directly.

The WorkHub MCP V1 server exposes 32 tools. It supports project, milestone, member, link, work item, work log, action item, fact-package, and overview workflows. It never exposes delete tools.

## Non-Negotiable Rules

1. Never commit `.env`, database files, Hermes config files, Feishu secrets, or real tokens.
2. Keep `HERMES_WORKHUB_TOKEN` identical in WorkHub and the Hermes MCP registration.
3. Do not let Hermes write to WorkHub until `health` and read-only tools pass.
4. The Feishu bot must use the same Hermes runtime/profile in which `workhub` MCP is registered.
5. Do not add delete tools or turn WorkHub into a full Feishu/Jira mirror.
6. Preserve the existing automatic-log boundary: only tracked work-item updates create system change logs. Important decisions, risks, blockers, and conclusions use explicit work logs.

## Recommended Topology

Deploy Hermes and WorkHub on the same internal host whenever possible.

```text
Hermes host: WorkHub at http://127.0.0.1:3000
Hermes MCP:  WORKHUB_BASE_URL=http://127.0.0.1:3000
```

If Hermes and WorkHub are on different hosts, use an internal HTTPS URL instead. `127.0.0.1` always means the Hermes host itself, not the user's laptop and not a Feishu cloud service.

## Prerequisites

- Git
- Node.js version compatible with the lockfile
- npm
- Hermes CLI installed on the host that receives Feishu requests
- A company-internal way to keep the WorkHub service running
- A private token generated by the company operator

For Windows + WSL deployments, Hermes runs inside WSL while WorkHub can run on Windows. The current scripts support that arrangement.

## First Deployment

### 1. Get the Code

```powershell
git clone <COMPANY_WORKHUB_REPOSITORY_URL> D:\workhub
cd D:\workhub
npm.cmd ci
```

### 2. Create the Environment File

```powershell
Copy-Item .env.example .env
```

Set a strong random value in `.env`:

```text
DATABASE_URL="file:./dev.db"
HERMES_WORKHUB_TOKEN="<GENERATE_A_LONG_RANDOM_COMPANY_SECRET>"
```

Keep this file on the deployment host only. Do not paste the token into Feishu, a prompt, a ticket, a document, or GitHub.

### 3. Initialize or Upgrade the Database

For a new environment:

```powershell
npx prisma db push
```

For an existing environment, back up the database before running schema updates. With the default SQLite setup, back up the configured `.db` file while WorkHub is stopped.

### 4. Build and Start WorkHub

```powershell
npm.cmd run typecheck
npm.cmd run build
npm.cmd run start
```

The default service address is `http://127.0.0.1:3000`.

Do not start a second server on the same port. If port 3000 is already occupied, identify whether the existing process is the intended WorkHub service before stopping it.

## Register the Hermes MCP Server

Run this from PowerShell, replacing placeholders with company values:

```powershell
wsl.exe -d <WSL_DISTRO> -u <WSL_USER> sh -lc "hermes mcp remove workhub || true"
wsl.exe -d <WSL_DISTRO> -u <WSL_USER> sh -lc "hermes mcp add workhub --command node --args '/mnt/d/workhub/scripts/hermes-workhub-mcp.mjs' --env WORKHUB_BASE_URL=http://127.0.0.1:3000 HERMES_WORKHUB_TOKEN=<THE_SAME_COMPANY_SECRET>"
```

For a non-WSL Linux host, run the equivalent commands directly and use the Linux repository path.

Hermes saves this configuration outside the repository, normally in `~/.hermes/config.yaml`. It must be recreated on every new deployment host.

Start a new Hermes session after changing MCP configuration.

## Technical Verification

First verify that Hermes discovers exactly 32 tools:

```powershell
wsl.exe -d <WSL_DISTRO> -u <WSL_USER> sh -lc "hermes mcp test workhub"
```

Then verify authenticated WorkHub access. Replace the token placeholder locally; do not send it to other people.

```powershell
Invoke-RestMethod `
  -Uri "http://127.0.0.1:3000/api/integrations/hermes/workhub" `
  -Method Post `
  -Headers @{ Authorization = "Bearer <THE_SAME_COMPANY_SECRET>" } `
  -ContentType "application/json" `
  -Body '{"tool":"health"}'
```

Expected result:

```text
service: workhub-hermes-v1
tools: 32 entries, including health
```

## Feishu Acceptance Prompts

Use a new Feishu/Hermes session. Run read-only checks first.

```text
请只调用 WorkHub 的 health 工具，并原样返回工具响应 JSON；不要自行统计、不要解释。
```

```text
请调用 WorkHub 的 get_workhub_overview，告诉我当前项目数、阻塞事项数、风险事项数和未完成 action items 数；不要创建或修改任何内容。
```

Then use a disposable test project to verify writing:

```text
请先搜索 WorkHub 是否已有“飞书Hermes MCP验证项目-YYYYMMDD”。如果没有，再创建该项目，并返回项目 ID。
```

```text
请在“飞书Hermes MCP验证项目-YYYYMMDD”中创建事项“V1.0 联通验证”，并创建三个 action items。返回事项和 action item 的 ID。
```

```text
请先读取“V1.0 联通验证”事项。确认唯一后，将状态更新为 following，并更新下一步行动。完成后确认是否生成了“事项变化”系统日志。
```

```text
请在该测试项目创建一条可汇报的决策日志，type=decision，source=feishu，reportable=true。
```

## Feishu Binding Check

Passing `hermes mcp test workhub` proves only the local Hermes CLI profile is correct. It does not prove that the Feishu bot uses that profile.

The Feishu bot is correctly connected only when the raw Feishu `health` response says:

```json
{
  "ok": true,
  "service": "workhub-hermes-v1",
  "tools": ["health", "..."]
}
```

If Feishu returns an older service name, a different tool count, or `401`, inspect the Hermes process used by Feishu. It may be a different user, machine, session, or MCP configuration.

## Hermes Agent Rules

Copy the WorkHub rules from `docs/hermes-workhub-v1.md` into the Hermes agent's system prompt or WorkHub skill. The minimum operational rules are:

```text
Use WorkHub MCP for all WorkHub reads and writes.
Search and identify the project before writing.
If needsConfirmation=true, ask the user to choose an ID before continuing.
Use YYYY-MM-DD dates. Do not guess missing project, owner, date, or status.
Never delete WorkHub data.
Use an explicit project log for decisions, risks, blockers, meeting conclusions, and reportable records.
Summarize only facts returned by WorkHub tools.
```

## Upgrade Procedure

1. Back up the database.
2. Stop the running WorkHub service.
3. Pull the approved Git revision.
4. Run `npm.cmd ci` if the lockfile changed.
5. Run `npx prisma db push` only when schema changes are part of the revision.
6. Run `npm.cmd run typecheck` and `npm.cmd run build`.
7. Start WorkHub.
8. Run `hermes mcp test workhub` and the raw `health` prompt.
9. Run the read-only overview prompt before allowing writes.

## Rollback Procedure

1. Stop WorkHub.
2. Check out the previously approved Git revision.
3. Restore the database backup only if a schema/data rollback is necessary.
4. Rebuild and start WorkHub.
5. Re-run `health` and a read-only overview check.

## Instructions for an AI Deploying This Repository

```text
Read AGENTS.md and docs/company-deployment.md before taking action.
Treat .env, database files, Feishu credentials, Hermes config, and tokens as local secrets; never add them to Git.
Deploy WorkHub and Hermes on the same internal host when possible.
Use the same HERMES_WORKHUB_TOKEN value in WorkHub and Hermes.
Do not modify MCP tool semantics, delete behavior, or automatic work-item log behavior during deployment.
Do not claim Feishu is connected until the Feishu bot itself returns the raw workhub health JSON from workhub-hermes-v1.
```
